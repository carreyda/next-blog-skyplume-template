## 1. 项目概述

### 1.1 这是什么

**Skyplume** 是一个开源的、内容优先（content-first）的个人博客模板，从作者 Ruoduan 的生产站点 [0x1ai.com](https://www.0x1ai.com/) 提炼而来，打包为可复用的开源起点。

它保留了严肃个人站点真正需要的部分：极速页面、干净的排版、强劲的 SEO 默认值、MDX 写作流程、RSS、站点地图、本地搜索、标签、暗色模式、克制的动画细节，以及一套易于定制的架构。

- 生产参考站点：<https://www.0x1ai.com/>
- 仓库：<https://github.com/ruoduan-hub/next-blog-skyplume-template>
- 一键部署：<https://vercel.com/new/clone?repository-url=https%3A%2F%2Fgithub.com%2Fruoduan-hub%2Fnext-blog-skyplume-template>
- 许可证：MIT

### 1.2 设计哲学

- **阅读优先于装饰**：界面采用克制的间距、锐利的字体、安静的边框与聚焦的文章版式，让写作始终在前。动画刻意短小轻盈，为导航和列表增添质感，而非把博客变成动画演示。
- **可被拥有的模板**：绝大多数定制从普通数据文件、MDX 文件和小型 React 组件开始，而非庞大的 CMS 或隐藏的配置层。贯彻「数据与 UI 分离」——修改配置文件即可完成个性化，无需触碰组件代码。
- **零配置即可启动**：评论、分析、订阅等可选功能在缺失凭证时静默降级而非报错，开发者 clone 后即可 `yarn dev` 启动。

### 1.3 核心特性一览

| 领域 | 包含内容 |
| --- | --- |
| 出版 | 来自 `data/blog` 的 MDX 文章、草稿、摘要、图片、分类、标签、作者、阅读时间、目录 |
| 内容渲染 | GitHub 风格 Markdown、KaTeX 数学公式、引用文献、代码标题、语法高亮、标题锚点、自定义图片与 iframe |
| 导航 | 首页 Feed、博客索引、分页、标签页、项目页、关于页、上一篇/下一篇 |
| SEO | App Router 元数据、每页 metadata 辅助函数、Open Graph、Twitter 卡片、JSON-LD `BlogPosting`、规范 URL、sitemap、robots、RSS |
| 发现 | 构建期 `search.json` 本地搜索、RSS feed、标签计数、结构化归档页 |
| 体验 | 响应式布局、暗色模式、本地搜索、评论、回到顶部/评论快捷键、进度条 |
| 性能 | 静态生成、Contentlayer 构建产物、字体优化加载、静态资源缓存头、可选静态导出 |
| 安全 | CSP、Referrer-Policy、框架保护、内容类型保护、HSTS、Permissions-Policy |

基准模板已调优至 Lighthouse 性能 / 最佳实践 / SEO 三项 100 分。

## 3. 快速开始

### 3.1 环境要求

- **Node.js**：22.x（由 `package.json` 的 `volta.node` 锁定为 22.23.1）
- **包管理器**：Yarn 3.6.1（仓库内置 `.yarn/releases/yarn-3.6.1.cjs`，无需全局安装）

> 若已安装 [Volta](https://volta.sh/)，进入项目目录后会自动切换到锁定的 Node 与 Yarn 版本。

### 3.2 安装与本地开发

```bash
yarn install   # 安装依赖
yarn dev       # 启动开发服务器
```

打开 <http://localhost:3000>。

评论、分析、订阅均为可选，模板在无任何凭证时即可正常启动。如需启用 Giscus 评论：

```bash
cp .env.example .env.local
# 然后在 .env.local 中填入 Giscus 配置（见第 15 章）
```

### 3.3 可用脚本

| 脚本 | 命令 | 说明 |
| --- | --- | --- |
| `start` | `next dev` | 启动开发服务器 |
| `dev` | `cross-env INIT_CWD=$PWD next dev` | 同上，额外设置 `INIT_CWD` 以兼容 Contentlayer 在 Windows 上的路径解析 |
| `build` | `cross-env INIT_CWD=$PWD next build && cross-env NODE_OPTIONS='--experimental-json-modules' node ./scripts/postbuild.mjs` | 生产构建（含 Contentlayer 生成）+ 运行 postbuild（生成 RSS） |
| `serve` | `next start` | 启动生产服务器（需先 build） |
| `analyze` | `cross-env ANALYZE=true next build` | 启用 `@next/bundle-analyzer` 的构建分析 |
| `test:local` | `node --test tests/*.test.mjs` | Node 原生 test runner 运行测试 |
| `lint` | `next lint --dir app --dir components --dir lib --dir layouts --dir scripts` | 对 5 个核心目录运行 ESLint |
| `prepare` | `husky` | `yarn install` 后自动执行，初始化 Git hooks |

`build` 中的 `NODE_OPTIONS='--experimental-json-modules'` 是必需的——因为 `scripts/rss.mjs` 使用了 `import tagData from '../app/tag-data.json' with { type: 'json' }` 导入断言语法。

### 3.4 首次配置清单

优先编辑这些文件：

- [data/siteMetadata.js](data/siteMetadata.js)：站点标题、URL、作者、邮箱、GitHub、评论、搜索、SEO 默认值
- [data/profile.ts](data/profile.ts)：关于页个人资料、链接、技能、模板指南内容
- [data/authors/default.mdx](data/authors/default.mdx)：默认作者元数据与简介
- [data/headerNavLinks.ts](data/headerNavLinks.ts)：顶部导航
- [data/projectsData.ts](data/projectsData.ts)：项目页内容
- `public/static/images/skyplume-card.svg`：社交分享卡片
- `public/static/favicons/favicon.svg`：站点图标（由 [scripts/generate-favicons.mjs](scripts/generate-favicons.mjs) 从 [data/logo.svg](data/logo.svg) 批量生成）

## 5. 整体架构与数据流

### 5.1 分层架构

Skyplume 在逻辑上可分为五层，自下而上依次为：

```text
┌──────────────────────────────────────────────────────────┐
│  内容层  data/*.mdx + data/*.bib + data/*.ts/.js          │ ← 写作的起点
├──────────────────────────────────────────────────────────┤
│  构建层  Contentlayer（文档类型 + Remark/Rehype 插件管道）  │ ← 编译期
│           + scripts/postbuild.mjs + scripts/rss.mjs        │
├──────────────────────────────────────────────────────────┤
│  数据管道  lib/content（sortPosts / allCoreContent）        │ ← 排序/瘦身/草稿过滤
│           lib/search（评分核心）                            │
├──────────────────────────────────────────────────────────┤
│  路由与布局  app/*（App Router）+ layouts/*                 │ ← 服务端组件取数
├──────────────────────────────────────────────────────────┤
│  表现层  components/*（业务组件 + shadcn/ui + 动画 + 主题）   │ ← 客户端交互
└──────────────────────────────────────────────────────────┘
```

贯穿所有层的是 [data/siteMetadata.js](data/siteMetadata.js)——它是站点的「中央配置枢纽」，被 28+ 文件消费，驱动 SEO 元数据、构建期产物路径与功能开关。

### 5.2 构建期数据流

构建（`yarn build`）发生的事情按顺序：

1. **Contentlayer 编译**（`next build` 内由 `withContentlayer` 触发）
   - 读取 `data/blog/*.mdx` 与 `data/authors/*.mdx`
   - 应用 6 个 Remark 插件 + 7 个 Rehype 插件
   - 计算派生字段：`readingTime` / `slug` / `path` / `filePath` / `toc` / `structuredData`
   - 输出到 `.contentlayer/generated/`（类型安全的 `allBlogs` / `allAuthors`）
2. **Contentlayer `onSuccess` 钩子**
   - `createTagCount(allBlogs)` → 写入 [app/tag-data.json](app/tag-data.json)（标签计数，`github-slugger` 标准化，`prettier` 格式化，生产环境过滤草稿）
   - `createSearchIndex(allBlogs)` → 当 `siteMetadata.search.provider === 'kbar'` 时写入 `public/search.json`（`allCoreContent(sortPosts(allBlogs))` 的精简结果）
3. **Next.js 构建** → 生成 `.next/`、`sitemap.xml`、`robots.txt`（来自 [app/sitemap.ts](app/sitemap.ts) / [app/robots.ts](app/robots.ts)，均声明 `force-static`）
4. **postbuild.mjs**（`next build` 后由 `&&` 串行触发）
   - 调用 [scripts/rss.mjs](scripts/rss.mjs)：从 `.contentlayer/generated` 读取 `allBlogs`，过滤草稿、按日期降序，生成主 `feed.xml`，并遍历 `tag-data.json` 为每个标签生成 `tags/<tag>/feed.xml`
   - 输出目录由 `process.env.EXPORT` 决定（静态导出写 `out/`，常规写 `public/`）

### 5.3 运行时渲染链路

以一篇博客文章为例，从请求到渲染：

```text
浏览器请求 /blog/<slug>
  → app/blog/[...slug]/page.tsx（服务端组件）
      ├─ generateStaticParams：构建期已预渲染所有文章
      ├─ generateMetadata：从 allBlogs 查文章 → 构造 article 级 OG/Twitter
      ├─ 取数据：allCoreContent(sortPosts(allBlogs)) → 计算 prev/next
      ├─ 注入 JSON-LD：<script type="application/ld+json">（post.structuredData + 作者）
      ├─ 选布局：layouts[post.layout || 'PostLayout']
      └─ <MDXLayoutRenderer code={post.body.code} components={components} toc={post.toc} />
            → lib/mdx/mdx-layout-renderer.tsx
                ├─ getMDXComponent：new Function() 把 code 字符串编译为 React 组件
                └─ <Mdx components={...} toc={...} />（注入自定义组件映射）
```

列表页、标签页、首页遵循相同的「服务端组件取数 → 传递 `CoreContent` 给展示组件」模式。

## 6. App Router 路由系统

项目采用 Next.js 15 App Router，以文件系统约定定义全部路由。除 [app/layout.tsx](app/layout.tsx) 与 [app/theme-providers.tsx](app/theme-providers.tsx) 为客户端组件外，其余页面均为服务端组件，在构建时通过 Contentlayer 从 MDX 生成静态数据。

整个 `app/` 目录只有一个根布局——`blog/` 与 `tags/` 子目录没有额外的 `layout.tsx`，所有页面共享同一套 Header / Footer / ThemeProvider / ProgressBar / Analytics 全局 Shell。

### 6.1 路由树

| 路由 | 文件 | 说明 |
| --- | --- | --- |
| `/` | [app/page.tsx](app/page.tsx) + [app/Main.tsx](app/Main.tsx) | 首页，展示最新 5 篇文章 |
| `/blog` | [app/blog/page.tsx](app/blog/page.tsx) | 博客列表第 1 页 |
| `/blog/page/[page]` | [app/blog/page/[page]/page.tsx](app/blog/page/[page]/page.tsx) | 博客列表分页 |

### 6.3 页面职责详解

**首页（数据获取与展示分离）**

[app/page.tsx](app/page.tsx) 是 async 服务端组件，负责取数：

```tsx
export default async function Page() {
  const sortedPosts = sortPosts(allBlogs)
  const posts = allCoreContent(sortedPosts)
  return <Main posts={posts} />
}
```

数据流：`allBlogs` → `sortPosts()`（按 `date` 降序，原地排序）→ `allCoreContent()`（剥离 `body`/`_raw`/`_id`，生产环境过滤草稿）→ 传入 [app/Main.tsx](app/Main.tsx)。

[app/Main.tsx](app/Main.tsx) 负责渲染：`MAX_DISPLAY = 5`，渲染顺序为 `<LabHero postCount={posts.length}>` → `Latest Posts` 标题 → `<PostFeed posts={displayPosts}>` → 若总数 > 5 显示 `All Posts` 链接 → 若配置了 `siteMetadata.newsletter?.provider` 则渲染 `<NewsletterForm />`。

**博客详情页（catch-all + 静态参数 + 动态元数据）**

[app/blog/[...slug]/page.tsx](app/blog/[...slug]/page.tsx) 关键实现：

- `generateStaticParams`：将每篇文章的 slug（如 `2024/my-post`）按 `/` 分割为字符串数组，构建期预渲染所有文章。
- `generateMetadata`：从 URL 拼出 slug 查文章；提取 `publishedTime`/`modifiedTime`（基于 `date`/`lastmod`）与作者；图片若不含 `http` 前缀则拼接 `siteMetadata.siteUrl` 构建 `og:images`；OpenGraph 类型设为 `article`。
- 渲染：若 `postIndex === -1`（草稿过滤导致）调用 `notFound()`；计算 `prev`/`next`；注入 JSON-LD；动态选择布局 `layouts[post.layout || defaultLayout]`；通过 `<MDXLayoutRenderer>` 渲染 MDX 正文。

**博客列表与分页**

[app/blog/page.tsx](app/blog/page.tsx)（第 1 页）与 [app/blog/page/[page]/page.tsx](app/blog/page/[page]/page.tsx)（分页）共用 `POSTS_PER_PAGE = 5` 与 `ListLayoutWithTags` 组件。分页页通过 `generateStaticParams` 预渲染所有分页路径，并对 `pageNumber <= 0`、`> totalPages`、`NaN` 调用 `notFound()` 返回 404。

**标签系统**

数据源是构建期生成的 [app/tag-data.json](app/tag-data.json)（`Record<string, number>`，标签名→文章数）。标签名经 `github-slugger` 的 `slug()` 标准化，确保中英文标签均可作为路由参数（`Open Source` → `open-source`，`开源` → `开源`）。

[app/tags/page.tsx](app/tags/page.tsx) 渲染统计面板、胶囊式标签按钮、Top topics 区域，以及一个独立的「Theme color philosophy」区块（展示 Mineral Teal 设计原则与色阶，相当于内嵌的风格指南）。标签过滤页 [app/tags/[tag]/page.tsx](app/tags/[tag]/page.tsx) 通过 `allBlogs.filter(post => post.tags.map(t => slug(t)).includes(tag))` 筛选，并在 `alternates` 中提供按标签的 RSS feed。

**自定义 404**

[app/not-found.tsx](app/not-found.tsx) 提供移动端纵向堆叠、桌面端横向排列（404 数字与说明文字用竖线 `md:border-r-2` 分隔）的 404 页面。两处触发：Next.js 自动路由未匹配，以及代码中显式 `notFound()`（文章不存在、页码无效）。

**SEO 辅助 `genPageMetadata`**

[app/seo.tsx](app/seo.tsx) 导出 `genPageMetadata({ title, description, image, ...rest })`，为各页面统一生成 Metadata：

```tsx
export function genPageMetadata({ title, description, image, ...rest }: PageSEOProps): Metadata {
  return {
    title,
    description: description || siteMetadata.description,
    openGraph: {
      title: `${title} | ${siteMetadata.title}`,
      description: description || siteMetadata.description,
      url: './',
      siteName: siteMetadata.title,
      images: image ? [image] : [siteMetadata.socialBanner],
      locale: 'zh_CN',
      type: 'website',
    },
    twitter: { title: `${title} | ${siteMetadata.title}`, card: 'summary_large_image', images: image ? [image] : [siteMetadata.socialBanner] },
    ...rest,   // 允许调用方覆盖/扩展任意字段
  }
}
```

关键设计：`description` 缺省回退到站点默认；OpenGraph `title` 自动拼接站点标题而 `title` 本身不拼接（与根 layout 的 `title.template` 配合）；图片优先级为自定义 `image` > `siteMetadata.socialBanner`；`...rest` 允许追加 `alternates`、`robots` 等。

> 注意：`genPageMetadata` 的 `locale` 硬编码为 `'zh_CN'`，与根布局 `openGraph.locale` 的 `'en_US'`、`siteMetadata.language` 的 `'en-us'` 不一致，属于模板的遗留配置，定制时可按需统一。

## 7. 内容系统（Contentlayer）

Skyplume 使用 Contentlayer2 构建完整的内容管理：声明式 frontmatter 管理文章与作者元数据，并通过一套 Remark/Rehype 插件管道将 MDX 源文件编译为可执行的 React 组件代码。配置集中在 [contentlayer.config.ts](contentlayer.config.ts)。

### 7.1 文档类型定义

通过 `makeSource({ contentDirPath: 'data', documentTypes: [Blog, Authors], ... })` 定义两种文档类型：

- **Blog**：匹配 `'blog/**/*.mdx'`，`contentType: 'mdx'`，博客文章核心类型。
- **Authors**：匹配 `'authors/**/*.mdx'`，`contentType: 'mdx'`，作者信息。

Contentlayer 在 `.contentlayer/generated/` 下生成类型安全的导出（`allBlogs` / `allAuthors` / `Blog` / `Authors`），由 [tsconfig.json](tsconfig.json) 的 `paths` 别名 `contentlayer/generated` → `.contentlayer/generated` 暴露。

### 7.2 Frontmatter 字段规范

**Blog 文档类型字段：**

| 字段 | 类型 | 必填 | 默认值 | 含义 |
| --- | --- | --- | --- | --- |
| `title` | string | 是 | — | 文章标题 |
| `date` | date | 是 | — | 发布日期（ISO 字符串） |
| `tags` | list\<string\> | 否 | `[]` | 标签列表，用于标签页、搜索、RSS |
| `categories` | list\<string\> | 否 | `[]` | 分类列表 |
| `lastmod` | date | 否 | — | 最后修改日期，用于 SEO `dateModified` |
| `draft` | boolean | 否 | — | 草稿标记，生产构建排除 |
| `summary` | string | 否 | — | 摘要，用于列表卡片、SEO description、RSS |
| `images` | json | 否 | — | 配图数组，首张作 Open Graph 图片 |
| `authors` | list\<string\> | 否 | — | 作者标识符，对应 `data/authors/` 文件名 |
| `layout` | string | 否 | — | 自定义布局名（覆盖默认 PostLayout） |
| `bibliography` | string | 否 | — | BibTeX 文件路径，用于学术引用 |
| `canonicalUrl` | string | 否 | — | 规范 URL，用于 SEO 去重 |

**Authors 文档类型字段：**

| 字段 | 类型 | 必填 | 含义 |
| --- | --- | --- | --- |
| `name` | string | 是 | 作者显示名称 |
| `avatar` | string | 否 | 头像图片路径 |
| `occupation` | string | 否 | 职业/身份 |
| `company` | string | 否 | 所属公司/组织 |
| `email` | string | 否 | 电子邮箱 |
| `twitter` / `bluesky` / `linkedin` / `github` | string | 否 | 社交主页链接 |
| `layout` | string | 否 | 布局名（如 `AuthorLayout`） |

Authors 的 MDX 正文为自由格式的作者简介。

**示例文章 frontmatter（[data/blog/term-proxy.mdx](data/blog/term-proxy.mdx)）：**

```mdx
---
title: Term Proxy — 告别手改 shell profile，用 Tauri 2 打造跨平台终端代理管理工具
date: '2026-06-28'
tags:
  - 开源
  - Tauri
  - Rust
  - React
categories:
  - 开源
draft: false
summary: >-
  屏幕右下角一个安静的小托盘，让你从此告别手改 .zshrc 切换代理的日子。
  基于 Tauri 2 + React 19 构建，支持 macOS、Windows、Linux，MIT 开源。

  - default
---
```

写法要点：`date` 用 ISO 格式且加引号避免 YAML 歧义；`summary` 用 `>-` 折叠块标量实现多行；`tags` 支持中文标签（`github-slugger` 原样保留中文 slug）；`authors: [default]` 对应 [data/authors/default.mdx](data/authors/default.mdx)。

### 7.3 计算字段（computedFields）

`computedFields` 对象（[contentlayer.config.ts](contentlayer.config.ts) 第 45-60 行）被 Blog 与 Authors 通过 `...computedFields` 共享复用：

| 计算字段 | 类型 | resolve 逻辑 |
| --- | --- | --- |
| `readingTime` | json | `readingTime(doc.body.raw)` → `{ text, minutes, time, words }` |
| `slug` | string | `doc._raw.flattenedPath.replace(/^.+?(\/)/, '')`（去掉 `blog/` 前缀） |
| `path` | string | `doc._raw.flattenedPath`（完整相对路径） |
| `filePath` | string | `doc._raw.sourceFilePath`（源文件完整路径） |
| `toc` | json | `extractTocHeadings(doc.body.raw)` → 标题数组 `{ value, url, depth }` |

Blog 额外定义 `structuredData`（类型 json），生成 Schema.org `BlogPosting` 结构化数据：

```typescript
structuredData: {
  type: 'json',
  resolve: (doc) => ({
    '@context': 'https://schema.org',
    '@type': 'BlogPosting',
    headline: doc.title,
    datePublished: doc.date,
    dateModified: doc.lastmod || doc.date,
    description: doc.summary,
    image: doc.images ? doc.images[0] : siteMetadata.socialBanner,
    url: `${siteMetadata.siteUrl}/${doc._raw.flattenedPath}`,
  }),
}
```

### 7.4 构建期副作用

`makeSource` 的 `onSuccess` 异步钩子在 Contentlayer 完成所有文档处理后触发，执行两个任务：

**`createTagCount(allBlogs)`：**

1. 遍历所有文章，对每篇的 `tags` 用 `github-slugger` 的 `slug()` 标准化
2. 计数累加（生产环境跳过 `draft: true`）
3. 用 `prettier.format()` 格式化后写入 [app/tag-data.json](app/tag-data.json)

当前生成结果示例：`{ "nextjs": 1, "mdx": 1, "seo": 1, "open-source": 1, "开源": 1, "tauri": 1, "rust": 1, "react": 1 }`

**`createSearchIndex(allBlogs)`：**

当 `siteMetadata.search.provider === 'kbar'` 且配置了 `searchDocumentsPath` 时，将 `allCoreContent(sortPosts(allBlogs))` 序列化为 JSON 写入 `public/search.json`（已被 `.gitignore` 忽略）。

### 7.5 内容工具函数

[lib/content/core.mjs](lib/content/core.mjs) 是内容数据流的核心工具模块：

| 函数 | 说明 |
| --- | --- |
| `sortPosts(allBlogs, dateKey='date')` | 对 ISO 日期字符串做字典序降序比较（等价于时间顺序，避免时区漂移），**原地排序**修改输入数组 |
| `coreContent(content)` | 通过 `omit` 剥离 `body`（编译后 MDX，体积大）、`_raw`、`_id`，返回可序列化的精简内容 |
| `allCoreContent(contents)` | 对数组逐个调用 `coreContent`；生产环境额外过滤 `draft === true`，开发环境保留草稿供预览 |
| `pick(obj, keys)` / `omit(obj, keys)` | 对象字段选择/剔除工具 |
| `sortedBlogPost(allBlogs)` | 向后兼容别名，内部调用 `sortPosts` |

标准数据流水线：`allCoreContent(sortPosts(allBlogs))`——先排序、再瘦身、生产环境过滤草稿，这一组合在搜索索引与 RSS 生成中复用。

辅助模块：

- [lib/content/format-date.mjs](lib/content/format-date.mjs)：`formatDate(date, locale='en-US')` → `"March 10, 2025"` 格式。
- [lib/content/html-escape.mjs](lib/content/html-escape.mjs)：`escapeHtml(value)` 转义 5 个 XML 不安全字符（`& < > ' "`），`String(value)` 强制转换保证对非字符串输入也安全；导出别名 `escape` 供 RSS 脚本使用。
- [lib/content/types.ts](lib/content/types.ts)：`export type CoreContent<T> = Omit<T, 'body' | '_raw' | '_id'>`——泛型工具类型，在类型层面确保列表/Feed 页面不会意外拉入编译后的大体积 MDX 数据。

## 8. MDX 处理管道

项目基于 Contentlayer2 构建了一条完整的 MDX 处理管道：从 `data/` 目录的 `.mdx` 源文件出发，经 Remark（Markdown AST 阶段）与 Rehype（HTML AST 阶段）插件处理，最终由运行时渲染器编译为 React 组件并注入自定义组件映射。

### 8.1 插件链总览

所有插件在 [contentlayer.config.ts](contentlayer.config.ts) 第 156-181 行的 `mdx.remarkPlugins` 与 `mdx.rehypePlugins` 数组中按序注册：

```text
Remark（mdast）：remarkExtractFrontmatter → remarkGfm → remarkCodeTitles
                → remarkMath → remarkImgToJsx → remarkAlert

Rehype（hast）： rehypeSlug → rehypeAutolinkHeadings → rehypeKatex
                → rehypeKatexNoTranslate → rehypeCitation
                → rehypePrismPlus → rehypePresetMinify
```

Contentlayer2 在 mdast 阶段应用 Remark 插件、在 hast 阶段应用 Rehype 插件，最终输出可执行的 MDX 函数源代码字符串存入 `post.body.code`。

### 8.2 Remark 插件

| 插件 | 来源 | 作用 |
| --- | --- | --- |
| `remarkExtractFrontmatter` | 自定义（[lib/mdx/plugins.mjs](lib/mdx/plugins.mjs)） | 遍历 mdast 的 `yaml` 节点，用 `gray-matter` 解析后写入 `file.data.frontmatter`，为统一生态工具链提供兼容 |
| `remarkGfm` | 社区 | GitHub 风格 Markdown：表格、删除线、任务列表、自动链接 |
| `remarkCodeTitles` | 自定义 | 支持 `` ```ts:lib/example.ts `` 语法，将冒号后的标题生成为 `<div className="remark-code-title">` 兄弟节点插入代码块前，并重置 `code` 节点的 `lang` |
| `remarkMath` | 社区 | 解析 `$...$`（行内）与 `$$...$$`（块级）数学公式 |
| `remarkImgToJsx` | 自定义 | 将 `public/` 下的本地 Markdown 图片转换为带 `width`/`height` 的 MDX `<Image>` 组件（二进制读取 PNG/JPEG/SVG 真实尺寸，避免 CLS）；同时将包裹的 `<p>` 改为 `<div>` 避免无效嵌套 |
| `remarkAlert` | `remark-github-blockquote-alert` | 将 `> [!NOTE]`、`> [!WARNING]` 等 GitHub 风格引用块转为带样式的提示块 |
| `remarkTocHeadings` / `extractTocHeadings` | 自定义 | 遍历 `heading` 节点，用 `github-slugger` 生成锚点 ID，收集为 `{ value, url, depth }` 数组 |

`remarkCodeTitles` 的核心实现——保持 AST 结构不变，仅插入兄弟节点：

```js
parent.children.splice(index, 0, {
  type: 'mdxJsxFlowElement',
  name: 'div',
  attributes: [{ type: 'mdxJsxAttribute', name: 'className', value: 'remark-code-title' }],
  children: [{ type: 'text', value: title }],
  data: { _xdmExplicitJsx: true },
})
node.lang = language   // 重置 code 节点语言为纯语言名
```

### 8.3 Rehype 插件

| 插件 | 配置 | 作用 |
| --- | --- | --- |
| `rehypeSlug` | — | 为 h1-h6 添加基于 `github-slugger` 的 `id` 锚点 |
| `rehypeAutolinkHeadings` | `behavior: 'prepend'`，`className: ['content-header']`，`content` 为 Heroicon 链接 SVG | 在标题前添加可点击的锚点链接图标（SVG 通过 `hast-util-from-html-isomorphic` 在构建期解析为 HAST 节点） |
| `rehypeKatex` | — | 将数学公式节点渲染为 KaTeX HTML |
| `rehypeKatexNoTranslate` | — | 为 KaTeX 元素添加 `translate="no"`，防止浏览器翻译干扰公式 |
| `rehypeCitation` | `{ path: path.join(root, 'data') }` | 从 `data/references-data.bib` 读取 BibTeX，支持 `[@Nash1950]` Pandoc 风格引用 |
| `rehypePrismPlus` | `{ defaultLanguage: 'js', ignoreMissing: true }` | Prism 语法高亮（行号、行高亮、diff），缺失语言不报错 |
| `rehypePresetMinify` | — | HTML 压缩：去除多余空白、合并文本节点、优化属性 |

**BibTeX 参考文献**：[data/references-data.bib](data/references-data.bib) 包含 4 条记录（Nash1950、Nash1951、Macfarlane2006、Xie2016）。文章可通过 `bibliography` frontmatter 字段指定专属 `.bib` 文件覆盖默认。

### 8.4 自定义 MDX 组件

[components/MDXComponents.tsx](components/MDXComponents.tsx) 是中央组件映射表，将原生 HTML 元素与自定义组件绑定到 MDX 渲染上下文：

```ts
export const components: MDXComponents = {
  Image,              // next/image 封装，自动处理 BASE_PATH
  TOCInline,          // 文章内嵌目录
  a: CustomLink,      // 智能链接（内部/锚点/外链）
  img: ProseImage,    // 混合图片渲染
  iframe: ProseIframe,// 嵌入式内容容器
  pre: Pre,           // 代码块（悬停复制）
  table: TableWrapper,// 表格横向滚动包装
  BlogNewsletterForm, // 可内联插入的订阅表单
}
```

> 模板未覆盖 `h1`-`h6` 与 `code`：标题由 `rehypeAutolinkHeadings` 在构建期生成链接图标，内联代码与代码块由 `rehypePrismPlus` + `Pre` 分别处理。

各 MDX 组件职责：

- **[components/Link.tsx](components/Link.tsx)（CustomLink）**：内部链接（`/` 开头）用 `next/link` 客户端路由；锚点（`#` 开头）用原生 `<a>`；外链用原生 `<a>` + `target="_blank"` + `rel="noopener noreferrer"`；统一 `break-words` 防溢出。
- **[components/mdx/Pre.tsx](components/mdx/Pre.tsx)**：`'use client'` 组件，悬停显示复制按钮，`navigator.clipboard.writeText` 复制，成功后切换为对勾图标 2 秒。
- **[components/mdx/ProseImage.tsx](components/mdx/ProseImage.tsx)**：混合策略——已知 `width`/`height` 用 `next/image`（优化+懒加载），未知回退原生 `<img>` + `decoding="async"`；支持 `data-zoom` 控制宽度。
- **[components/mdx/ProseIframe.tsx](components/mdx/ProseIframe.tsx)**：`<iframe>` 包裹在 `overflow-hidden` 容器中防移动端溢出，默认 `title="Embedded content"`、`loading="lazy"`。
- **[components/mdx/TOCInline.tsx](components/mdx/TOCInline.tsx)**：接收 `toc` 数组，用 `createNestedList`（栈算法）将扁平 TOC 重建为嵌套树；支持 `fromHeading`/`toHeading`（层级范围）、`exclude`（正则排除）、`asDisclosure`（`<details>` 折叠模式）。
- **[components/mdx/Bleed.tsx](components/mdx/Bleed.tsx)**：全宽出血布局，`full` 模式用 `mr-[calc(-50vw+50%)] ml-[calc(-50vw+50%)]` 实现视口全宽，非 full 模式用 `-mx-6 md:-mx-8 2xl:-mx-24`。

### 8.5 运行时渲染

[lib/mdx/mdx-layout-renderer.tsx](lib/mdx/mdx-layout-renderer.tsx) 负责将 Contentlayer 生成的 MDX 代码字符串在运行时编译为 React 组件：

- **`getMDXComponent(code, globals)`**：用 `new Function(...)` 动态构造接收 `React`、`ReactDOM`、`_jsx_runtime` 与自定义 `globals` 的函数，返回模块 `default` 导出组件。此设计避免运行时 Babel/MDX 编译器的体积开销。
- **`useMDXComponent(code, globals)`**：`React.useMemo` 包装 `getMDXComponent`，按 `code`/`globals` 缓存编译结果。
- **`MDXLayoutRenderer`**：接收 `code` 与 `components`，渲染 `<Mdx components={components} {...rest} />`，将 `toc` 等剩余 props 透传给 MDX 组件。

完整链路：`data/blog/*.mdx` →（构建期插件）→ `post.body.code` + `post.toc` →（[app/blog/[...slug]/page.tsx](app/blog/[...slug]/page.tsx)）`<MDXLayoutRenderer code={post.body.code} components={components} toc={post.toc} />` → 运行时编译渲染。

## 9. 布局系统

Skyplume 提供三种文章详情布局与一种列表布局，覆盖不同视觉风格需求。

### 9.1 三种文章布局

布局通过 frontmatter 的 `layout` 字段在 [app/blog/[...slug]/page.tsx](app/blog/[...slug]/page.tsx) 运行时动态选择，未声明时默认回退 `PostLayout`：

```tsx
const defaultLayout = 'PostLayout'
const layouts = { PostSimple, PostLayout, PostBanner }
const Layout = layouts[post.layout || defaultLayout]
```

**差异对比：**

| 特性 | PostLayout（默认） | PostBanner | PostSimple |
| --- | --- | --- | --- |
| 作者信息 | 展示 | 不展示 | 不展示 |
| 标签 | 侧边栏展示 | 不展示 | 不展示 |
| 日期格式 | 含星期（如 `Monday, March 10, 2025`） | 不展示日期 | 不含星期（如 `March 10, 2025`） |
| 横幅图片 | 无 | 有（Bleed 全宽，回退 picsum 占位图） | 无 |
| 布局结构 | 双栏（`lg:grid lg:grid-cols-4`，侧边栏+正文） | 单栏通栏 | 单栏通栏 |
| 上一篇/下一篇 | 卡片式（带边框背景，`line-clamp-2`） | 内联文本链接 | 内联文本链接 |
| Back to posts | 有 | 无 | 无 |
| 评论 | 有（可配置） | 有（可配置） | 有（可配置） |

**适用场景**：PostLayout 适合内容丰富的技术文章、多作者博客；PostBanner 适合图文并茂的视觉类博客；PostSimple 适合极简主义博客、个人日记、快速笔记。使用者只需在文章 frontmatter 声明 `layout: PostSimple` 或 `layout: PostBanner` 即可切换。

三种布局均集成：

- **ScrollTopAndComment** 浮动按钮（[components/ScrollTopAndComment.tsx](components/ScrollTopAndComment.tsx)）：固定右下角，滚动超 50px 显示，提供「回到顶部」与「跳转评论」（仅当配置了 `comments.provider`）两个按钮；退出采用 200ms 延迟卸载策略保证退出动画完整。
- **Comments 评论**：懒加载模式，用户点击 `Load comments` 才注入 Giscus 脚本。
- **上一篇/下一篇**：在页面中按 `allCoreContent(sortPosts(allBlogs))` 计算 `prev`/`next`，每对象仅传 `{ path, title }`。

### 9.2 列表布局

[layouts/ListLayoutWithTags.tsx](layouts/ListLayoutWithTags.tsx) 是唯一的列表布局组件，被 `/blog`、`/blog/page/[page]`、`/tags/[tag]`、`/tags/[tag]/page/[page]` 四个路由共用。

- **`'use client'` 组件**：使用 `usePathname()` 判断当前活跃路径以高亮标签与 `All Posts` 链接。
- **标签云侧栏**：数据源为 [app/tag-data.json](app/tag-data.json)，按文章数降序；当前选中标签以 `text-primary-500` 高亮为不可点击的 `<h3>`，未选中为指向 `/tags/{slug(tag)}` 的 `<Link>`；固定宽度 `max-w-[280px] min-w-[280px]`、可滚动、带阴影。
- **文章列表**：每条展示日期、标题链接、标签、摘要；列表项带 `animate-item-reveal` 入场动画，延迟递增 `index * 40ms`。
- **分页（内置 `Pagination` 组件）**：展示 `Previous | 当前页/总页数 | Next`，URL 规则为第 1 页 `/{basePath}/`、第 N 页 `/{basePath}/page/{N}`，使用 `rel="prev"`/`rel="next"` 语义化链接。

Props 接口：

```tsx
interface ListLayoutProps {
  posts: CoreContent<Blog>[]             // 全量（用于标签统计）
  title: string
  initialDisplayPosts?: CoreContent<Blog>[]  // 当前页显示
  pagination?: PaginationProps
}
```

### 10.2 shadcn/ui 基础组件层

[components/ui/](components/ui/) 下的 6 个组件遵循统一的 shadcn/ui 设计模式：

- **无头组件基元**：使用 `@base-ui/react`（替代传统 shadcn 的 Radix UI），提供无障碍、键盘交互、焦点管理，不强制样式。
- **变体系统**：`class-variance-authority`（cva）定义 `variants` 与 `defaultVariants`。
- **类名合并**：通过 `cn()`（clsx + tailwind-merge）合并 cva 变体类名与外部 `className`，确保用户可覆盖默认样式。
- **`data-slot` 语义标记**：如 `data-slot="button"`、`data-slot="dialog-content"`，便于 Tailwind `**` 深度选择器。
- **变体导出**：如 `export { buttonVariants }`，允许其他组件通过 cva 派生或组合样式。

**Button**（[components/ui/button.tsx](components/ui/button.tsx)）：基于 `@base-ui/react/button` 的 `ButtonPrimitive`。6 个 variant（default/outline/secondary/ghost/destructive/link）× 10 个 size（default `h-8` / xs / sm / lg / icon / icon-xs / icon-sm / icon-lg）。通过 `render` prop 支持 polymorphic 渲染，`nativeButton={!render}` 确保无自定义 render 时渲染原生 button。

**Badge**（[components/ui/badge.tsx](components/ui/badge.tsx)）：与 Button 不同，使用 `@base-ui/react/use-render` + `merge-props`（默认渲染 `<span>`，可 `render` prop 改为其他元素）。6 个 variant，胶囊形状 `rounded-4xl`。

**Accordion**（[components/ui/accordion.tsx](components/ui/accordion.tsx)）：复合组件模式，导出 4 个子组件。通过 Tailwind `group-aria-expanded/accordion-trigger:hidden` 模式**纯 CSS** 切换展开/折叠图标，无需 JavaScript；内容面板用 `data-open:animate-accordion-down data-closed:animate-accordion-up` 自定义关键帧动画。

**Dialog**（[components/ui/dialog.tsx](components/ui/dialog.tsx)）：导出 10 个子组件（Dialog/DialogTrigger/DialogPortal/DialogClose/DialogOverlay/DialogContent/DialogHeader/DialogFooter/DialogTitle/DialogDescription）。DialogContent 默认居中（`top-1/2 left-1/2 -translate-x-1/2 -translate-y-1/2`），`w-[min(calc(100%-2rem),32rem)]`，动画通过 `data-starting-style`/`data-ending-style` 声明式定义（`scale-95 opacity-0`）。

**Input**（[components/ui/input.tsx](components/ui/input.tsx)）：原生 `<input>` 样式化包装，`h-9 w-full`，支持 focus-visible 环、selection 高亮、disabled、aria-invalid 破坏性样式。

**Separator**（[components/ui/separator.tsx](components/ui/separator.tsx)）：基于 `@base-ui/react/separator`，`orientation` prop（horizontal/vertical），通过 data 属性驱动样式。

### 10.3 支撑层

[lib/utils.ts](lib/utils.ts) 导出 `cn` 函数，是整个 UI 组件体系的基石：

```ts
import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'
export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

`clsx` 处理条件类名、数组、对象合并；`twMerge` 在此基础上解决 Tailwind 类名冲突（后者覆盖前者，如 `twMerge('px-4 py-2 px-6')` → `'py-2 px-6'`），正确处理 Tailwind 4 任意值语法。

## 11. 样式系统

### 11.1 Tailwind CSS 4 CSS-first 配置

项目使用 Tailwind CSS 4 的 CSS-first 配置方式，**无需 `tailwind.config.js`**，所有设置通过 CSS 完成。

[postcss.config.js](postcss.config.js) 仅声明 `@tailwindcss/postcss` 插件（v4 标准 PostCSS 集成）：

```js
module.exports = { plugins: { '@tailwindcss/postcss': {} } }
```

[css/tailwind.css](css/tailwind.css) 的导入与插件声明层级：

```css
@import './animations.css';     /* 自定义 CSS 动画 */
@import 'tailwindcss';          /* Tailwind CSS v4 核心 */
@import 'tw-animate-css';       /* 第三方动画工具类库 */
@import 'shadcn/tailwind.css';  /* shadcn/ui v4 基础样式 */

@plugin "@tailwindcss/forms";     /* 表单元素样式重置 */
@plugin '@tailwindcss/typography'; /* prose 排版工具类 */
```

### 11.2 Mineral Teal 主题色与 OKLCH

主题色名为 **Mineral Teal**（矿物青），在注释中描述为「calm lab-grade accent with teal-tinted neutrals」。所有颜色使用 **OKLCH 色彩空间**定义——L（明度）、C（色度）、H（色相）三通道独立控制，感知均匀，暗模式下不会发灰。

**Primary 色板（11 阶）**，色相统一在 178-186 度（青/矿物青）：

| Token | OKLCH | 用途 |
| --- | --- | --- |
| `--color-primary-50` | `oklch(0.972 0.012 180)` | 极浅 |
| `--color-primary-200` | `oklch(0.89 0.052 180)` | 低色度，柔和 |
| `--color-primary-400` | `oklch(0.69 0.115 179)` | **暗模式主色**，色度最高 |
| `--color-primary-500` | `oklch(0.56 0.12 178)` | **亮模式主色** |
| `--color-primary-700` | `oklch(0.42 0.095 180)` | 亮模式内联代码 |
| `--color-primary-950` | `oklch(0.22 0.045 186)` | 极深 |

**Gray 中性灰板（11 阶）**：色相统一在 190 度（略带青色调），与 primary 形成协调氛围，避免纯中性灰的「死板」感。

颜色通过 `@theme` 块定义为 `--color-primary-*`、`--color-gray-*`，配合自定义行高 `--line-height-11`~`14`、扩展 z-index `--z-60/70/80`。

### 11.3 明暗双模

采用两层 CSS 变量架构：

**第一层 `:root` 语义变量**（亮色模式）：

```css
:root {
  --background: oklch(1 0 0);
  --foreground: oklch(0.145 0 0);
  --primary: var(--color-primary-600);
  --primary-foreground: oklch(0.985 0 0);
  --secondary: var(--color-gray-100);
  --muted: var(--color-gray-100);
  --accent: var(--color-primary-50);
  --border: var(--color-gray-200);
  --ring: var(--color-primary-500);
  --radius: 0.625rem;
  /* ... sidebar、chart 等 ... */
}
```

**第二层 `.dark` 覆写**（暗色模式）：`--background` 切到 `--color-gray-950`，`--primary` 切到 `--color-primary-400`（暗背景上需更亮以保持对比度），`--border` 用 `oklch(1 0 0 / 10%)` 半透明白色。

**暗模式激活机制**：通过 `@custom-variant dark (&:where(.dark, .dark *))` 定义，当 `<html>` 有 `.dark` 类时所有 `dark:` 变体生效（比 v3 默认的 `@media (prefers-color-scheme)` 更灵活，允许手动切换）。这套机制由 `next-themes` 的 `attribute="class"` 驱动。

**shadcn/ui v4 桥接**：`@theme inline` 块将语义化 CSS 变量桥接到 Tailwind 工具类（`--color-primary: var(--primary)` → `bg-primary`），不重复生成 CSS 自定义属性。[components.json](components.json) 配置 `style: "base-nova"`、`rsc: true`、`cssVariables: true`、`baseColor: "neutral"`、`iconLibrary: "lucide"`。

### 11.4 Typography 与 Prism

**Typography Prose 覆盖**：通过 unlayered（非层叠）CSS 规则精细控制 `@tailwindcss/typography` 的 prose 样式——这些规则必须放在 `@layer` 外部以确保最高级联优先级，否则会被插件默认值覆盖。亮模式定义正文 `--tw-prose-body`、标题、链接（`var(--color-primary-500)`）、内联代码等；暗模式使用 `.dark .prose` 选择器（而非 `prose-invert`）**完全绕过**插件的 `.prose-invert` 编译期拦截，所有属性使用 `!important` 确保覆盖。

**Prism 语法高亮**：采用 **rehype-prism-plus** 在 MDX 编译期生成静态 Prism token HTML，运行时仅加载 [css/prism.css](css/prism.css)。该文件首行 `@reference "./tailwind.css"` 声明引用（允许用 `@apply` 但不产生重复输出），定义代码块结构样式（`.remark-code-title`、`.code-line`、`.highlight-line`、`.line-number::before`）与 **Night Owl** token 主题（Sarah Drasner，MIT）。

**Base 层全局样式**：v3-v4 边框兼容（显式设 `border-color: var(--color-gray-200, currentColor)` 恢复 v3 行为）、可访问性聚焦样式（`a:focus-visible`/`button:focus-visible` 使用 Mineral Teal `outline`）、shadcn 语义类映射（`* { @apply border-border }`、`body { @apply bg-background text-foreground }`）、WebKit 自动填充修复（`transition: background-color 600000s` 禁用黄色背景）。

**字体**：Space Grotesk 通过 `next/font/google` 加载，CSS 变量 `--font-space-grotesk` 绑定到 `@theme` 的 `--font-sans` 与 `--font-heading`。这种方式无 FOUT，字体文件由 Next.js 自动子集化与预加载。

**[css.d.ts](css.d.ts)**：声明 `declare module '*.css' { const content: string; export default content }`，让 `import 'css/prism.css'` 等侧效导入通过 TypeScript 检查。

**[prettier.config.js](prettier.config.js)**：`semi: false`、`singleQuote: true`、`printWidth: 100`、`tabWidth: 2`、`trailingComma: 'es5'`、`plugins: ['prettier-plugin-tailwindcss']`（自动按推荐顺序排序 Tailwind 类名）。

## 12. 动画与交互系统

动画系统遵循「克制、轻量」的设计哲学：**完全不依赖** Framer Motion、GSAP、react-spring 等第三方动画库，所有动画通过 CSS `@keyframes`（集中在 [css/animations.css](css/animations.css)）或 Web Animations API 实现。

统一动画语言：`opacity + translateY` 的淡入上浮，缓动曲线统一为 `cubic-bezier(0.25, 0.1, 0.25, 1)`，持续时间 80ms–500ms。所有动画组件均为 `'use client'`（依赖 `IntersectionObserver`、`Element.animate()`、`window.scrollY` 等浏览器 API）。

[css/animations.css](css/animations.css) 定义 4 个 `@keyframes`，位移量从 4px → 16px → 20px → 24px 递增，对应不同场景的视觉权重：

| 关键帧 | 位移 | 时长 | CSS 类 | 场景 |
| --- | --- | --- | --- | --- |
| `page-enter` | 4px | 0.15s | `.page-transition` | 路由切换（最轻） |
| `item-reveal` | 16px | 0.35s | `.animate-item-reveal` | 列表项揭示 |
| `hero-reveal` | 20px | 0.45s | `.animate-hero-reveal` | 首屏英雄区 |
| `scroll-reveal` | 24px | 0.5s | （JS 动态绑定） | 滚动揭示（最重） |

所有动画仅操作 `opacity` 与 `transform`（GPU 合成属性），不引起 layout/paint 重计算。

**各动画组件：**

- **[components/PageTransition.tsx](components/PageTransition.tsx)**：利用 React `key={pathname}` 驱动——路由变化时 `pathname` 改变，React 卸载旧 `<div>` 挂载新的，重新触发 `.page-transition` 动画。
- **[components/ProgressBar.tsx](components/ProgressBar.tsx)**：唯一使用 Web Animations API 的组件。固定顶部 2px 进度条，路由变化时 5 阶段步进（`scaleX` 0.25→0.55→0.82→0.96→1，每阶段 `await animation.finished`），完成后 150ms 淡出；通过检查 `bar.getAnimations().length` 衔接多次快速切换。在根布局中被 `<Suspense fallback={null}>` 包裹（因使用 `useSearchParams()`），`fallback={null}` 避免阻塞整页。
- **[components/ScrollReveal.tsx](components/ScrollReveal.tsx)**：可复用包装组件，`IntersectionObserver`（`rootMargin: '-60px'`）监听，进入视口时动态设 `el.style.animation = 'scroll-reveal 0.5s ... forwards'`，触发后立即 `observer.disconnect()` 保证只触发一次。支持 `delay` prop。在 AboutPage 大量使用实现区块依次揭示。
- **[components/home/LabHero.tsx](components/home/LabHero.tsx)**：首页英雄区，纯 CSS 交错揭示——所有子元素共享 `.animate-hero-reveal` 类，通过递增 `animationDelay`（0ms→80ms→160ms→240ms）依次出现。
- **[components/home/PostFeed.tsx](components/home/PostFeed.tsx)**：每个 `PostFeedItem` 独立 `IntersectionObserver`（`rootMargin: '-40px'`），进入视口时添加 `.animate-item-reveal` 并设 `animationDelay = index * 60ms`。
- **[layouts/ListLayoutWithTags.tsx](layouts/ListLayoutWithTags.tsx)**：直接用 `.animate-item-reveal` + `animationDelay: index * 40ms`（纯 CSS 触发，非 IntersectionObserver），适合列表页期望所有可见项依次出现。

**三种动画触发模式：**

1. **CSS class 自动触发**（页面加载即播）：LabHero 的 hero-reveal，通过 `animationDelay` 交错，无需 JS。
2. **IntersectionObserver 按需触发**（进入视口时播）：PostFeed、ScrollReveal，`disconnect()` 确保仅触发一次。
3. **React key 重新挂载触发**（路由变化时播）：PageTransition。

动画与主题系统完全解耦——`@keyframes` 只操作 `opacity`/`transform`，配色由 Tailwind `dark:` 变体在 `<html>` 层级一次性完成，无主题切换闪烁问题。

## 13. 本地搜索系统

本地搜索是模板最近一次重构（commit `9580236`：*refactor: migrate search to shadcn/ui and extract scored-search core*）的核心功能，采用「构建期预生成静态索引 + 客户端纯内存评分排序」架构，不依赖任何外部搜索服务。

### 13.1 评分核心（纯函数）

[lib/search/core.mjs](lib/search/core.mjs) 导出两个零依赖的纯函数 `matchesSearchQuery` 与 `filterSearchDocuments`，采用**四级字段权重**评分体系：

| 字段命中 | 分数 |
| --- | --- |
| 标题（title） | 4 |
| 标签（tags） | 3 |
| 摘要（summary） | 2 |
| 路径（path） | 1 |
| 无匹配 | 0 |

匹配通过 `value.toLowerCase().includes(query)` 实现大小写不敏感的部分包含。`filterSearchDocuments` 采用 **map → filter → sort → slice → map** 管道：

```js
export function filterSearchDocuments(documents, query, limit = DEFAULT_LIMIT) {
  const normalizedQuery = normalizeQuery(query)
  if (!normalizedQuery) return documents.slice(0, limit)   // 空查询返回前 8 条预览

  return documents
    .map((document, index) => ({ document, index, score: scoreDocument(document, normalizedQuery) }))
    .filter((result) => result.score > 0)
    .sort((a, b) => b.score - a.score || a.index - b.index)  // 同分按原始顺序稳定排序
    .slice(0, limit)                                          // DEFAULT_LIMIT = 8
    .map((result) => result.document)
}
```

空查询时直接返回前 8 篇（按日期降序的最新文章）作为 `Recent posts` 初始预览。

### 13.2 索引生成

搜索索引在 Contentlayer 构建成功后的 `onSuccess` 钩子中由 `createSearchIndex` 生成：当 `siteMetadata.search.provider === 'kbar'` 且配置了 `searchDocumentsPath` 时，将 `allCoreContent(sortPosts(allBlogs))` 序列化写入 `public/search.json`。`allCoreContent` 剥除 `body`/`_raw`/`_id`、`sortPosts` 按日期降序、生产环境过滤草稿。

### 13.3 SearchProvider 与对话框 UI

[components/search/SearchProvider.tsx](components/search/SearchProvider.tsx) 是 React Context 提供者，重构后全面采用 shadcn/ui 组件体系：

- **状态管理**：`isOpen`（对话框开关）、`query`（输入）、`documents`（索引）、`isLoading`/`hasLoaded`/`error`（异步三态）。
- **响应式筛选**：`useMemo` 调用 `filterSearchDocuments(documents, query)`。
- **竞态防护**：`requestIdRef` 递增计数器 + `AbortController` 双重机制——每次发起新请求 `requestIdRef.current++`，`.then()` 回调通过闭包捕获的 `requestId` 与当前 ref 比对，确保只有最新请求结果写入状态；对话框关闭时 cleanup 调用 `controller.abort()`。
- **UI**：`Dialog` + `DialogContent`（自定义 className 覆盖居中为顶部偏移 `top-6`/`sm:top-16`，宽度 `w-[min(calc(100%-1rem),44rem)]`）；搜索输入用 shadcn `Input`（`type="search"`，带搜索图标装饰，关联 `aria-describedby`）；结果区 `max-h-[min(60dvh,30rem)] overflow-y-auto`、`aria-live="polite"`。
- **辅助组件**：`SearchLoadingRows`（3 行骨架屏 + Loader2 旋转图标）、`SearchState`（统一处理未配置/加载失败/无结果）、`SearchResultButton`（FileText 图标 + 标题 + 日期/路径 + 摘要 + 最多 2 个标签 Badge + 悬停滑入的 ArrowRight）。

### 13.4 键盘快捷键与触发入口

全局 `keydown` 注册两个快捷键：**Cmd/Ctrl + K** 打开（`preventDefault` 阻止 Chrome DevTools 等默认行为）、**Escape** 关闭并清空输入；对话框打开后 `setTimeout(() => inputRef.current?.focus(), 0)` 自动聚焦。

[components/SearchButton.tsx](components/SearchButton.tsx) 是触发入口，检查 `siteMetadata.search?.provider === 'kbar'` 决定是否渲染（否则返回 `null`）；按钮用 shadcn `Button`（`variant="ghost"`、`size="icon-lg"`）+ lucide `Search` 图标 + `sr-only` 的 `Search` 文本标签。

### 13.5 优雅降级的 Context 设计

`useSearch()` hook 在 Context 为 `null` 时不抛异常，而是返回空操作的 `openSearch: () => undefined`——使 SearchButton 等消费者在无 SearchProvider 包裹时也能安全渲染。`openSearch` 用 `useCallback`（空依赖）保证引用稳定。

### 13.6 重构改进要点

相比重构前：① 匹配算法从「拼接单一 haystack 做 `includes`」升级为四级评分排序；② UI 从手写固定定位 div + `role="dialog"` 迁移到 shadcn/ui Dialog/Input/Button/Badge 组件体系；③ 新增骨架屏、结构化空状态、结果计数标签、底部状态栏；④ 搜索逻辑抽取为可独立测试的纯函数模块。

单元测试 [tests/search-core.test.mjs](tests/search-core.test.mjs)（Node 原生 `node:test`）覆盖：空查询返回前 8 条预览；标题匹配（4 分）排在标签匹配（3 分）之前。

## 14. SEO 与内容发现

SEO 体系由四个层面构成：App Router 原生约定路由（sitemap/robots）构建期输出静态文件；`genPageMetadata` 辅助函数统一注入 OG/Twitter 卡片；文章页通过 `structuredData` 计算字段生成 JSON-LD；postbuild 脚本生成 RSS。所有默认值收敛于 [data/siteMetadata.js](data/siteMetadata.js)。

### 14.1 Sitemap 与 Robots

[app/sitemap.ts](app/sitemap.ts) 声明 `dynamic = 'force-static'`，合并两类路由：

- **静态路由**：`''`（首页）、`blog`、`projects`、`tags`，`lastModified` 用构建当天日期。
- **博客路由**：过滤 `draft: true`，取 `post.path`，`lastModified` 优先 `post.lastmod` 兜底 `post.date`。

[app/robots.ts](app/robots.ts) 同样 `force-static`：`userAgent: '*'` + `allow: '/'`、`sitemap` 指向 `${siteUrl}/sitemap.xml`、`host` 设为 `siteUrl`。

### 14.2 文章级 generateMetadata

[app/blog/[...slug]/page.tsx](app/blog/[...slug]/page.tsx) 的 `generateMetadata` 不依赖 `genPageMetadata`，独立构建文章级 Metadata：`publishedAt`/`modifiedAt` 来自 `date`/`lastmod`；`authors` 从 `authorDetails` 提取；`ogImages` 优先 `post.images`（`img.includes('http')` 判断，相对路径拼 `siteUrl`）；`openGraph.type` 设为 `'article'`（区别于列表页的 `'website'`），携带 `publishedTime`/`modifiedTime`——这是 Google 展示文章丰富摘要的关键信号。

### 14.3 JSON-LD 结构化数据

两阶段生成：

1. **构建期**：`contentlayer.config.ts` 的 `structuredData` 计算字段返回 Schema.org `BlogPosting` 对象（`headline`/`datePublished`/`dateModified`/`description`/`image`/`url`）。
2. **运行期**：文章页从 `post.structuredData` 读取，注入动态 `author` 数组（`@type: 'Person'`），通过 `dangerouslySetInnerHTML` 内联为 `<script type="application/ld+json">`。

### 14.4 RSS 生成

[scripts/rss.mjs](scripts/rss.mjs) 在 `next build` 后由 [scripts/postbuild.mjs](scripts/postbuild.mjs) 触发：

- **输出目录**：`process.env.EXPORT` 决定写 `out/`（静态导出）或 `public/`。
- **主 feed**：过滤 `draft !== true`，`sortPosts` 降序，模板字符串拼接 RSS 2.0 XML。`<item>` 的 `title`/`description` 经 `escape()` 转义 XML 不安全字符，`pubDate` 转 UTC 字符串，`author` 格式 `${email} (${author})`，`<category>` 由 `tags` 映射。
- **按标签拆分**：遍历 `app/tag-data.json` 每个标签，筛选包含该标签的文章，分别生成 `tags/<tag>/feed.xml`（`mkdirSync({ recursive: true })` 自动建目录）。每个标签都有独立 RSS 订阅端点。

### 14.5 草稿过滤的多层一致性

项目在多处实现生产环境过滤草稿，保证开发可预览、生产不可见：

| 位置 | 过滤逻辑 |
| --- | --- |
| sitemap.ts | `!post.draft`（不区分环境） |
| rss.mjs | `post.draft !== true`（不区分环境） |
| createTagCount | `!isProduction \|\| file.draft !== true`（生产排除，开发保留） |
| allCoreContent | 生产过滤 `draft === true`，开发保留 |
| blog/[...slug] | 间接使用 allCoreContent，草稿在生产返回 404 |

### 14.6 OG/Twitter 卡片优先级链

```text
列表页/通用页（genPageMetadata）：
  og:title / twitter:title  = "<页面title> | <siteMetadata.title>"
  og:description            = 传入 description || siteMetadata.description
  og:image / twitter:image  = 传入 image || siteMetadata.socialBanner
  og:type                   = 'website'
  twitter:card              = 'summary_large_image'

文章详情页（独立 generateMetadata）：
  og:type                   = 'article'  ← 文章专用
  og:publishedTime/modifiedTime = frontmatter date / lastmod
  og:images                 = post.images（相对路径自动拼 siteUrl 前缀）

JSON-LD：
  image = doc.images[0] || siteMetadata.socialBanner
  url   = siteMetadata.siteUrl + doc._raw.flattenedPath
```

## 15. 评论 / 分析 / 订阅

Skyplume 将评论、站点分析、邮件订阅设计为**可选的增强能力**，并在缺失凭证时静默降级而非报错。「无凭证也能启动」是贯穿全局的设计原则。

| 功能 | 默认状态 | 降级行为 |
| --- | --- | --- |
| Giscus 评论 | `provider: 'giscus'` 但环境变量为空 | 显示配置引导提示，不影响页面渲染 |
| 分析 | `analytics: {}` | Analytics 组件返回 `null`，零脚本注入 |
| 订阅 | `newsletter.provider: ''` | 首页不渲染表单；API 返回 500 但不崩溃 |
| 搜索 | `provider: 'kbar'` 已配置 | 本地搜索不需外部服务，开箱即用 |

### 15.1 评论系统（Giscus）

**统一入口** [components/Comments.tsx](components/Comments.tsx)：接收 `{ slug, title }`，读 `siteMetadata.comments.provider`——仅当为 `'giscus'` 时渲染「Load comments」按钮（带 `MessageCircle` 图标，shadcn `Button` `variant="outline"`），点击后才挂载 `<GiscusComments>`（**懒加载**，避免首屏加载第三方脚本，也符合 GDPR 隐私合规）；否则返回 `null`。

**Giscus 实现** [components/comments/GiscusComments.tsx](components/comments/GiscusComments.tsx)（`'use client'`）：

1. **配置校验** `getGiscusConfig()`：从 4 个环境变量读取（`NEXT_PUBLIC_GISCUS_REPO`/`_REPOSITORY_ID`/`_CATEGORY`/`_CATEGORY_ID`）。任一缺失则 `missing` 数组记录，渲染友好提示引导到 giscus.app。
2. **脚本注入**（useEffect）：清空容器后创建 `<script src="https://giscus.app/client.js">`，通过 `dataset` 传配置（`data-mapping="pathname"`、`data-strict="0"`、`data-reactions-enabled="1"`、`data-input-position="bottom"`、`data-lang="en"`、`crossorigin="anonymous"`、`async`）。
3. **主题同步**：`next-themes` 的 `useTheme()` 获取 `resolvedTheme`，`'dark'` 映射 Giscus 主题 `'dark_dimmed'`，否则 `'light'`；主题变化时通过 `postMessage` 向 Giscus iframe（`'https://giscus.app'`）发送 `{ giscus: { setConfig: { theme } } }` 实时同步，无需重载 iframe。

**双层开关**：① 配置级——`siteMetadata.comments.provider` 设为空字符串彻底关闭；② 凭证级——保持 `provider: 'giscus'` 但环境变量缺失时显示提示。

**启用步骤**（见 [.env.example](.env.example) 注释）：GitHub 仓库设为公开并启用 Discussions → 安装 Giscus App → 在 giscus.app 生成配置填入 `.env.local`。

### 15.2 站点分析（Analytics 占位）

[components/analytics/Analytics.tsx](components/analytics/Analytics.tsx) 定义了完整的 `AnalyticsConfig` 类型，覆盖 6 种服务（Plausible/Simple Analytics/PostHog/Umami/Google Analytics/Microsoft Clarity），但组件函数体**仅 `return null`**——占位实现。它在根布局中被渲染 `<Analytics analyticsConfig={siteMetadata.analytics as AnalyticsConfig} />`。

[next.config.js](next.config.js) 的 CSP 已为 `analytics.umami.is` 预留 script-src 白名单，表明 Umami 是推荐的第一个接入选项。启用需在 Analytics.tsx 补充脚本注入逻辑（类似 Giscus 模式）并在 `siteMetadata.analytics` 填入配置。

### 15.3 邮件订阅（Newsletter）

完整链路由 4 个文件构成：

- **[components/newsletter/NewsletterForm.tsx](components/newsletter/NewsletterForm.tsx)**（`'use client'`）：核心表单，接收 `title` 与 `apiUrl`（默认 `/api/newsletter`）。`useRef` 管理输入值，提交时 `fetch` POST `{ email }`，成功后 `subscribed=true` 禁用输入（placeholder 变 `"You're subscribed"`，按钮变 `"Thank you!"`），错误显示红色提示。
- **[components/newsletter/BlogNewsletterForm.tsx](components/newsletter/BlogNewsletterForm.tsx)**：NewsletterForm 的展示包装器（灰色圆角容器），注册为全局 MDX 组件，作者可在任意 MDX 文章内联 `<BlogNewsletterForm />`。
- **[lib/newsletter/api.ts](lib/newsletter/api.ts)**：`NewsletterAPI(options)` 工厂函数返回 App Router handler。当前：无 email 返回 400 `{ error: 'Email is required' }`，有 email 返回 500 `{ error: '<provider> not supported' }`。
- **[app/api/newsletter/route.ts](app/api/newsletter/route.ts)**：调用工厂并 `export { handler as GET, handler as POST }`，声明 `dynamic = 'force-static'`——未配置时该路由静态化为返回固定 500 JSON 的端点，部署到任意静态托管平台均无问题，也不产生无意义的 Serverless 调用。

接入真实邮件服务（Buttondown/ConvertKit/Mailchimp）只需在 `lib/newsletter/api.ts` 增加 provider 分支、在 `siteMetadata` 填 provider 名 + API key（环境变量），**前端代码无需修改**——API 契约（POST `{ email }`，返回 `{ error }` 或成功）保持不变。

### 15.4 siteMetadata 作为功能开关

[data/siteMetadata.js](data/siteMetadata.js) 是中央配置枢纽，四大可选功能均通过它控制开关，且所有 provider 配置均为字符串/空对象、不含密钥——真正密钥全部通过环境变量注入（除 `NEXT_PUBLIC_` 前缀外不进入客户端 bundle）。

## 18. 部署

### 18.1 Vercel 部署

最快路径是 README 顶部的一键部署按钮（克隆仓库到 Git 提供商、创建 Vercel 项目、默认配置即可部署）。[vercel.json](vercel.json) 配置三类缓存 header：

- **非静态资源** `/((?!_next/static|static|feed\.xml|search\.json|favicon\.ico).*)`：`public, max-age=0, s-maxage=3600, stale-while-revalidate=86400, must-revalidate`（浏览器每次验证，CDN 缓存 1 小时，SWR 24 小时）。
- **`/_next/static/(.*)`**（含 content hash）：`public, max-age=31536000, immutable`（永久缓存一年）。
- **`/static/(.*)`**：同样一年不可变缓存。

### 18.2 静态导出

通过环境变量实现同一代码库在 Vercel SSR 与纯静态导出间切换：

```bash
EXPORT=true UNOPTIMIZED=true yarn build
# 部署到子路径时附加：
BASE_PATH=/my-sub-path yarn build
```

- `EXPORT=true` → `output: 'export'` 生成纯静态 HTML 到 `out/`。
- `UNOPTIMIZED=true` → 关闭 Next.js 内置图片优化（静态导出必须）。
- `BASE_PATH` → 所有静态资源路径（社交卡片、搜索索引、favicon、RSS feed）自动添加前缀。

### 18.3 安全响应头

[next.config.js](next.config.js) 的 `headers()` 应用到所有路由（`/((.*))`）与 `/static/:path*`：

| Header | 值 | 作用 |
| --- | --- | --- |
| Content-Security-Policy | `default-src 'self'; script-src 'self' 'unsafe-eval' 'unsafe-inline' giscus.app analytics.umami.is; ...` | 默认 deny，精细开放 giscus.app（评论）、analytics.umami.is（分析）、music.163.com（音乐嵌入） |
| Referrer-Policy | `strict-origin-when-cross-origin` | 仅向同源发送完整 referrer |
| X-Frame-Options | `DENY` | 禁止被嵌入 iframe |
| X-Content-Type-Options | `nosniff` | 禁止 MIME 嗅探 |
| X-DNS-Prefetch-Control | `on` | 启用 DNS 预取 |
| Strict-Transport-Security | `max-age=31536000; includeSubDomains` | 强制 HTTPS |
| Permissions-Policy | `camera=(), microphone=(), geolocation=()` | 禁用敏感 API |

## 21. 设计亮点与技术细节

本项目在工程实现上有诸多值得借鉴的设计，汇总如下：

**内容与数据管道**

- Contentlayer 共享 `computedFields` 通过展开语法在 Blog/Authors 间复用，Blog 追加专属 `structuredData`，体现清晰的字段继承。
- `allCoreContent(sortPosts(allBlogs))` 构建标准数据流水线：排序 → 剥离 `body`/`_raw`/`_id` → 生产过滤草稿，在搜索索引与 RSS 中复用。
- `CoreContent<T>` 泛型在类型层面确保列表/Feed 页面不会意外拉入编译后的大体积 MDX 数据。
- 自研 `remarkImgToJsx` 二进制读取 PNG/JPEG/SVG 文件头获取真实尺寸，避免 CLS 且对不支持格式安全降级。
- `remarkCodeTitles` 用冒号语法将标题注入为兄弟 `mdxJsxFlowElement` 节点而非修改 code 节点，保持 AST 结构。
- `mdx-layout-renderer` 用 `new Function()` 动态编译而非运行时 MDX 编译器，极大降低客户端 JS 体积；`useMemo` 按 code 缓存避免重复编译。

**搜索系统**

- 搜索核心为纯函数模块，四级字段权重评分（标题 4 > 标签 3 > 摘要 2 > 路径 1），同分按原始顺序稳定排序。
- `requestIdRef` + `AbortController` 双重竞态防护；`useSearch` 在 Context 为 null 时优雅降级返回空操作函数。
- 重构将手写 div 迁移到 shadcn/ui Dialog 体系，获得结构化状态展示与统一无障碍标注。

**动画系统**

- 完全不依赖第三方库（零体积开销），仅 CSS `@keyframes` + Web Animations API。
- 4 个关键帧位移量 4px→24px 递增构成有层次的动画语言，统一缓动函数保证交互感受一致。
- PageTransition 用 React `key` + CSS class 极简实现路由切换动画；ScrollTopAndComment 用 entering/exiting 双状态机 + setTimeout 延迟卸载优雅处理退出动画。
- 动画与主题完全解耦——`@keyframes` 操作 opacity/transform，配色由 Tailwind `dark:` 变体独立处理。

**SEO**

- 文章页 `og:type` 设为 `'article'` 并携带 `publishedTime`/`modifiedTime`，区别于列表页 `'website'`，是 Google 文章丰富摘要的关键信号。
- JSON-LD 在构建期预计算，运行时仅注入动态 author 数组。
- RSS 不仅生成主 feed，还为每个标签生成独立 `tags/<tag>/feed.xml`，实现精细化订阅。
- `og:images` 自动检测 URL 是否以 `http` 开头决定是否拼接 `siteUrl` 前缀，符合 OG 协议对绝对 URL 的要求。

**样式系统**

- 全部颜色基于 OKLCH 色彩空间，明度变化线性，暗模式不发灰。
- Mineral Teal：11 阶 primary 色板 + 11 阶带 190 度青色调的中性灰，整体和谐。
- Typography 覆盖用 unlayered 规则 + `!important` 策略，完全绕过 `@tailwindcss/typography` 的默认样式控制。
- Prism 编译期生成，运行时零 JS 开销；Tailwind 4 CSS-first 配置无需 `tailwind.config.js`。

**工程化**

- ESLint 9 flat config 集成 TS/jsx-a11y/Prettier/Next 规则形成完整静态检查。
- Git 提交门禁三层防护：pre-commit（lint-staged）+ commit-msg（commitlint）。
- Node 原生 test runner 零额外框架依赖。
- 环境变量 `EXPORT`/`UNOPTIMIZED`/`BASE_PATH` 实现同一代码库在 Vercel SSR 与静态导出间切换。
- CSP 精细到具体第三方域，默认 deny 最小化攻击面。
- devcontainer 实现 Codespaces 一键启动。
- `generate-favicons` 从单一 SVG 源批量生成 10 个图标 + PWA manifest。

**可选功能的降级哲学**

所有可选功能遵循统一策略：环境变量缺失不报错（GiscusComments 用 try-get 收集缺失项渲染友好提示）；条件渲染而非条件导入；API 路由 `force-static`；CSP 预留白名单——确保 zero-config 即可 `yarn dev` 启动，同时保留完整的扩展点。
