## 1. 重点一：这个项目有「后端」吗？

### 1.1 一句话结论

> **没有传统意义的后端。** 本项目是一个**以静态生成为主（SSG）的前端项目**：没有数据库、没有 ORM、没有认证系统、没有运行时服务端业务逻辑、没有中间件（middleware）。唯一一个 API 路由 `/api/newsletter` 是一个**静态化的「契约占位」**，并不真正处理订阅。

这个结论是用代码逐项验证过的，下面给出证据。

### 1.2 把「疑似后端」的东西逐个掰开看

很多人看到「API 路由」「Server Component」就会以为是后端。我们逐个对照源码：

#### ① 唯一的 API 路由：`/api/newsletter`

文件 [app/api/newsletter/route.ts](app/api/newsletter/route.ts) 全部内容只有 11 行：

```ts
import { NewsletterAPI } from '@/lib/newsletter/api'
import siteMetadata from '@/data/siteMetadata'

export const dynamic = 'force-static'   // ← 关键：声明为静态

const handler = NewsletterAPI({
  // @ts-ignore
  provider: siteMetadata.newsletter.provider,   // 默认是空字符串 ''
})

export { handler as GET, handler as POST }
```

再看它调用的工厂 [lib/newsletter/api.ts](lib/newsletter/api.ts)，handler 的核心逻辑只有两条分支（第 34–38 行）：

```ts
if (!email) {
  return NextResponse.json({ error: 'Email is required' }, { status: 400 })
}
return NextResponse.json({ error: `${options.provider || ''} not supported` }, { status: 500 })
```

**结论**：收到邮箱后，它不写数据库、不调用邮件服务、不发请求，直接返回一个 `500 not supported`。这是一个**保留接口契约、但没有任何真实业务逻辑的占位实现**。

#### ② Server Components ≠ 后端

项目里大量页面是 Server Component（服务端组件），例如首页 [app/page.tsx](app/page.tsx)：

```tsx
export default async function Page() {
  const sortedPosts = sortPosts(allBlogs)      // allBlogs 来自构建期生成的数据
  const posts = allCoreContent(sortedPosts)
  return <Main posts={posts} />
}
```

这里的「服务端」指的是 **Next.js 在构建时（build time）渲染 HTML**，数据来源是 Contentlayer 在构建期编译好的 `allBlogs`——**不是在请求时去查数据库**。这属于「构建期取数 + 静态生成」，不是运行时后端。

#### ③ 构建期数据管道 ≠ 后端

项目在 `yarn build` 时会「生成」一些数据文件：

| 产物 | 由谁生成 | 性质 |
| --- | --- | --- |
| `app/tag-data.json`（标签计数） | [contentlayer.config.ts](contentlayer.config.ts) 的 `createTagCount`（`onSuccess` 钩子） | 构建期一次性生成，提交或部署时已是静态文件 |
| `public/search.json`（搜索索引） | 同上的 `createSearchIndex` | 构建期生成，前端 fetch |
| `feed.xml`（RSS） | [scripts/rss.mjs](scripts/rss.mjs)，由 [scripts/postbuild.mjs](scripts/postbuild.mjs) 在 build 后触发 | 构建期生成的静态 XML |

这些都是**编译/构建阶段的一次性产物**，不是运行时服务。部署后它们就是普通的静态文件，和一张 PNG 没有本质区别。

#### ④ 第三方服务：逻辑都在外部

| 能力 | 本地做了什么 | 真正逻辑在哪 |
| --- | --- | --- |
| 评论（Giscus） | 渲染一个「Load comments」按钮，点击后注入 `<script>` 加载 [giscus.app](https://giscus.app) | GitHub Discussions + Giscus 服务 |
| 站点分析 | [components/analytics/Analytics.tsx](components/analytics/Analytics.tsx) **函数体仅 `return null`**，是占位 | 未启用；推荐接 Umami（CSP 已预留白名单） |
| 邮件订阅 | 表单 POST 到 `/api/newsletter` | 未启用（见 ①，返回 500 占位） |

也就是说：评论、分析、订阅这些「需要后端」的能力，要么完全托管在第三方（Giscus），要么留了扩展点但当前不工作。

#### ⑤ 没有中间件、没有数据库、没有认证

对仓库全局检索可验证：

- **无 middleware**：项目根目录与 `app/` 下不存在 `middleware.ts`，也没有 `export function middleware`。
- **无数据库 / ORM / 认证库**：检索 `prisma / drizzle / mongoose / better-sqlite / @vercel/postgres / ioredis / upstash / next-auth / clerk / supabase` 等关键词，**项目源码中均无命中**（仅在 Yarn 自带的 `yarn-3.6.1.cjs` 里有无关字符串）。

### 1.3 渲染策略：它本质是个静态站点

把上面的证据合起来，项目的整体渲染策略是：

```text
                 ┌── 构建期（yarn build）──────────────────────┐
   data/*.mdx ──→│ Contentlayer 编译 → allBlogs / allAuthors    │
                 │ onSuccess → tag-data.json / search.json      │
                 │ next build → 预渲染所有页面（generateStaticParams）│
                 │ postbuild → feed.xml                         │
                 └──────────────────────────────────────────────┘
                                      ↓
                 ┌── 运行期（部署后）────────────────────────────┐
                 │ 全部是预渲染好的静态 HTML + 静态 JSON / XML     │
                 │ 没有数据库查询、没有运行时计算                  │
                 └──────────────────────────────────────────────┘
```

关键佐证：

- 文章详情页 [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 78 行导出了 `generateStaticParams`，**构建期就把每一篇文章预渲染成静态 HTML**。
- [app/sitemap.ts](app/sitemap.ts)、[app/robots.ts](app/robots.ts)、[app/api/newsletter/route.ts](app/api/newsletter/route.ts) 都显式声明了 `force-static`。
- 配置支持 `output: 'export'`（见 [next.config.js](next.config.js) 第 57 行），可一键导出成**纯静态文件**部署到任意静态托管（GitHub Pages、对象存储等），进一步印证「无需 Node 运行时」。

> **一句话**：能 `output: 'export'` 导出成纯静态文件、丢到任意静态托管上跑的项目，本质上就不是「有后端」的项目。

### 1.4 那个 /api/newsletter 到底是干嘛的

既然没有真实订阅逻辑，为什么还要留这个路由？这是模板的一个**精心设计**：

1. **保留 API 契约**：前端订阅表单 [components/newsletter/NewsletterForm.tsx](components/newsletter/NewsletterForm.tsx) 写死了 `fetch('/api/newsletter', { method:'POST', body:{ email } })`。留一个路由，前端代码不用改。
2. **声明 `force-static`**：在未配置 provider 时，这个路由会在构建时被静态化成一个固定响应的端点——**部署到静态托管也不会报错**，也不会产生无意义的 Serverless 调用。
3. **预留扩展点**：未来真要接邮件服务，只需在 [lib/newsletter/api.ts](lib/newsletter/api.ts) 里给 `NewsletterAPI` 增加 provider 分支（调用 Buttondown / ConvertKit / Mailchimp 等），并在 [data/siteMetadata.js](data/siteMetadata.js) 填上 provider 名与 API key（走环境变量）。**前端代码一行都不用动**。

所以它的定位是：**「接口先占住、后端后接」的脚手架**，而不是已有的后端服务。

### 1.5 当你真的需要后端时，怎么扩展

本项目留好了清晰的扩展点，按需启用即可：

| 你想要 | 怎么做 | 涉及文件 |
| --- | --- | --- |
| 真正的邮件订阅 | 给 `NewsletterAPI` 加 provider 分支，密钥用环境变量 | [lib/newsletter/api.ts](lib/newsletter/api.ts)、[data/siteMetadata.js](data/siteMetadata.js) |
| 站点分析 | 补充脚本注入逻辑（仿 Giscus 的 `<script>` 注入方式） | [components/analytics/Analytics.tsx](components/analytics/Analytics.tsx) |
| 运行时接口（如访问统计、留言保存） | 新建 `app/api/<your-route>/route.ts`，**不要声明 `force-static`**，并按需引入数据库 | 自行新增 |
| 数据库 | 自行引入 Prisma / Drizzle 等（本项目不自带，需自己加） | 自行新增 |

> 注意：一旦引入了**运行时动态数据**（数据库、按请求计算的接口），就不再是纯静态站点，`output: 'export'` 静态导出可能不再适用，需要部署到支持 Node 运行时的平台（如 Vercel）。

## 2. 重点二：如何新增一篇文章

### 2.1 30 秒最短路径

在 [data/blog/](data/blog/) 目录下新建一个 `.mdx` 文件，文件头写好 frontmatter，下面写正文，重启/触发热更新即可：

```mdx
---
title: 我的第一篇文章
date: '2026-06-29'
tags:
  - 随笔
summary: 这是一段出现在列表卡片和搜索引擎结果里的摘要。
authors:
  - default
---

正文从这里开始，支持所有 Markdown 与 MDX 能力。
```

文件名就是文章的 slug（URL）。例如 `data/blog/my-first-post.mdx` → 访问地址 `/blog/my-first-post`。

> 仓库里有两篇真实范例可参考：[data/blog/term-proxy.mdx](data/blog/term-proxy.mdx)、[data/blog/skyplume-blog-template.mdx](data/blog/skyplume-blog-template.mdx)。

### 2.2 从一个 .mdx 到全站可见：完整数据流

很多模板只告诉你「建个文件」，但不讲文章是怎么「自动」出现在首页、列表、标签页、搜索、RSS 里的。Skyplume 的完整链路如下：

```text
你新建 data/blog/foo.mdx
        │
        │  ① yarn dev / yarn build 触发
        ▼
Contentlayer 编译（contentlayer.config.ts）
   ├── 解析 frontmatter（title/date/tags…）
   ├── 跑 Remark + Rehype 插件链（GFM、数学、代码高亮、代码标题…）
   ├── 计算派生字段：readingTime / slug / path / toc / structuredData
   └── 输出类型安全的 allBlogs（.contentlayer/generated/）
        │
        ├──② onSuccess 钩子
        │     ├── createTagCount → app/tag-data.json     （标签计数 → 标签页 + 标签云）
        │     └── createSearchIndex → public/search.json （搜索索引 → 本地搜索）
        │
        │  ③ next build 预渲染
        ▼
各页面消费 allBlogs：
   ├── app/page.tsx           首页：sortPosts → allCoreContent → 取最新 5 篇
   ├── app/blog/page.tsx      列表：POSTS_PER_PAGE=5 分页
   ├── app/blog/[...slug]     详情：generateStaticParams 预渲染，generateMetadata 出 OG/JSON-LD
   ├── app/tags/*             标签：按 tag 过滤
   └── app/sitemap.ts         sitemap：自动收录（过滤草稿）
        │
        │  ④ postbuild.mjs → rss.mjs
        ▼
   public/feed.xml + tags/<tag>/feed.xml（RSS，每个标签都有独立订阅）
```

**要点**：你**只需要写一个 `.mdx` 文件**，排序、草稿过滤、标签统计、搜索索引、SEO 元数据、RSS、sitemap 全部由上面的管道**自动**完成。

### 2.3 frontmatter 字段全表

来自 [contentlayer.config.ts](contentlayer.config.ts) 第 96–113 行的 `Blog` 文档类型定义：

| 字段 | 类型 | 必填 | 默认 | 作用 / 影响范围 |
| --- | --- | --- | --- | --- |
| `title` | string | ✅ | — | 文章标题；用于 `<title>`、OG、JSON-LD `headline` |
| `date` | date | ✅ | — | 发布日期（建议加引号 `'2026-06-29'`）；用于**排序**、SEO `datePublished` |
| `tags` | list&lt;string&gt; | ❌ | `[]` | 标签页、标签云计数、搜索评分、RSS `<category>` |
| `categories` | list&lt;string&gt; | ❌ | `[]` | 分类（当前主要用于内容组织） |
| `lastmod` | date | ❌ | — | 最后修改日期；用于 SEO `dateModified` |
| `draft` | boolean | ❌ | — | **草稿标记**：开发可见、生产隐藏（见 2.8） |
| `summary` | string | ❌ | — | 列表卡片摘要、SEO `description`、搜索摘要 |
| `images` | json | ❌ | — | 配图数组，**首张作为 Open Graph 分享图** |
| `authors` | list&lt;string&gt; | ❌ | `['default']` | 作者标识，对应 [data/authors/](data/authors/) 下的文件名 |
| `layout` | string | ❌ | `'PostLayout'` | 切换布局：`PostSimple` / `PostLayout` / `PostBanner` |
| `bibliography` | string | ❌ | — | BibTeX 文件路径，用于学术引用 `[@key]` |
| `canonicalUrl` | string | ❌ | — | 规范 URL，用于 SEO 去重 |

**写法要点**：

- `date` 用 ISO 格式且**加引号**，避免 YAML 把它解析成日期对象导致歧义。
- `summary` 多行时用 `>-`（折叠块标量）。
- `tags` 支持中文（`github-slugger` 会原样保留中文 slug）。
- 派生字段（你不用写、自动生成）：`readingTime`（阅读时间）、`slug`、`path`、`toc`（目录）、`structuredData`（JSON-LD）。

### 2.4 正文里能用什么（MDX 能力清单）

正文默认支持标准 Markdown + GitHub 风格（表格、删除线、任务列表）。此外，[components/MDXComponents.tsx](components/MDXComponents.tsx) 注入了一批自定义组件与处理能力：

| 能力 | 在 mdx 里怎么写 | 由谁实现 |
| --- | --- | --- |
| 智能链接（内部走 `next/link`，外链新标签页） | `[文字](/path)` 或 `[文字](https://...)` | [components/Link.tsx](components/Link.tsx)（CustomLink） |
| 图片（本地自动转 `next/image` 并注入尺寸） | `![alt](/static/blog/foo/bar.png)` | remarkImgToJsx 插件 + [components/mdx/ProseImage.tsx](components/mdx/ProseImage.tsx) |
| 代码块（语法高亮 + 悬停复制） | ` ```js ... ``` ` | rehype-prism-plus + [components/mdx/Pre.tsx](components/mdx/Pre.tsx) |
| 代码块带文件名标题 | ` ```js:lib/example.js ... ``` `（冒号后是标题） | remarkCodeTitles 插件 |
| 行内/块级数学公式 | `$E=mc^2$` 或 `$$...$$` | remark-math + rehype-katex |
| GitHub 风格提示块 | `> [!NOTE]` / `[!TIP]` / `[!WARNING]` / `[!IMPORTANT]` / `[!CAUTION]` | remark-github-blockquote-alert |
| 学术引用 | `[@Nash1950]`（需配 `bibliography`） | rehype-citation + [data/references-data.bib](data/references-data.bib) |
| 标题锚点链接 | 任意 `## 标题` 自动生成可点击锚点 | rehype-slug + rehype-autolink-headings |
| 内联目录 | `<TOCInline toc={toc} />` | [components/mdx/TOCInline.tsx](components/mdx/TOCInline.tsx) |
| 全宽出血布局 | `<Bleed full>...</Bleed>` | [components/mdx/Bleed.tsx](components/mdx/Bleed.tsx) |
| 内联订阅表单 | `<BlogNewsletterForm />` | [components/newsletter/BlogNewsletterForm.tsx](components/newsletter/BlogNewsletterForm.tsx) |
| 嵌入 iframe | 标准 `<iframe>`（自动包裹防溢出） | [components/mdx/ProseIframe.tsx](components/mdx/ProseIframe.tsx) |

### 2.5 图片怎么放、怎么引用

- **存放位置**：建议放 `public/static/blog/<post-slug>/` 下，例如 `public/static/blog/my-first-post/cover.png`。
- **引用方式**：`![封面](/static/blog/my-first-post/cover.png)`（路径以 `/static/...` 开头，对应 `public/` 目录）。
- **尺寸自动注入**：自定义插件 `remarkImgToJsx` 会在构建期**二进制读取 PNG/JPEG/SVG 文件头**，拿到真实宽高注入 `next/image`，从而**避免布局抖动（CLS）**。
- **远程图片**：仅允许 `picsum.photos`（见 [next.config.js](next.config.js) 的 `images.remotePatterns`）；用其它图床需在配置里追加域名，或用 `UNOPTIMIZED=true` 关闭图片优化。
- **配图作分享图**：frontmatter 的 `images` 字段首张会作为 Open Graph / Twitter 卡片图。

### 2.6 作者怎么关联

1. 在 [data/authors/](data/authors/) 下新建或编辑一个 `.mdx`，文件名即作者标识（默认已有 [data/authors/default.mdx](data/authors/default.mdx)）。
2. 作者 frontmatter 字段：`name`（必填）、`avatar`、`occupation`、`company`、`email`、`github`/`twitter`/`bluesky`/`linkedin`、`layout`（通常 `AuthorLayout`）。正文是自由格式的作者简介。
3. 在文章 frontmatter 写 `authors: [default]`（或多作者 `[a, b]`）。
4. 文章详情页 [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 30、95 行会按 `authors` 字段查找作者详情并渲染；**未声明时自动回退到 `default`**（见第 30 行 `post?.authors || ['default']`）。

### 2.7 切换文章布局

在 frontmatter 加 `layout` 字段即可切换三种文章详情布局（默认 `PostLayout`）：

| 布局 | 写法 | 特点 |
| --- | --- | --- |
| `PostLayout`（默认） | 不写或 `layout: PostLayout` | 双栏：左侧侧边栏（作者 + 标签 + 目录）、右侧正文；日期含星期 |
| `PostBanner` | `layout: PostBanner` | 单栏 + 顶部全宽横幅大图；不显示作者/标签 |
| `PostSimple` | `layout: PostSimple` | 极简单栏；适合短笔记、日记 |

布局选择发生在 [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 17–22、109 行：

```tsx
const defaultLayout = 'PostLayout'
const layouts = { PostSimple, PostLayout, PostBanner }
// ...
const Layout = layouts[post.layout || defaultLayout]
```

### 2.8 草稿机制

在 frontmatter 写 `draft: true`，会得到「**开发可见、生产隐藏**」的行为：

| 环境 | 行为 |
| --- | --- |
| `yarn dev`（开发） | 草稿**可见**，方便预览 |
| `yarn build`（生产） | 草稿从首页、列表、标签计数、搜索索引、RSS、sitemap 中**全部排除**；直接访问草稿 URL 会 **404** |

这是因为草稿过滤在多处一致地实现（核心在 [lib/content/core.mjs](lib/content/core.mjs) 的 `allCoreContent`：生产环境过滤 `draft === true`；[contentlayer.config.ts](contentlayer.config.ts) 的 `createTagCount`、[scripts/rss.mjs](scripts/rss.mjs)、[app/sitemap.ts](app/sitemap.ts) 也各自过滤）。

## 3. 重点三：用到了哪些 Next.js 知识

本章把项目里**实际用到**的 Next.js 知识点分类盘点，每项说明「在本项目怎么用」和「为什么这么用」。理解这些，二次开发基本无障碍。

### 3.1 App Router 文件系统路由

Next.js 13+ 的 App Router 用**文件系统约定**定义路由，本项目全面采用：

| 约定文件 | 作用 | 本项目实例 |
| --- | --- | --- |
| `app/layout.tsx` | **唯一根布局**，全局 Shell（Header/Footer/主题/Provider） | [app/layout.tsx](app/layout.tsx) |
| `app/page.tsx` | 路由根页面 | 首页 [app/page.tsx](app/page.tsx) |
| `app/<dir>/page.tsx` | 子路由页面 | [app/blog/page.tsx](app/blog/page.tsx)、[app/about/page.tsx](app/about/page.tsx)、[app/projects/page.tsx](app/projects/page.tsx) |
| `app/<dir>/route.ts` | API 路由（Route Handler） | [app/api/newsletter/route.ts](app/api/newsletter/route.ts) |
| `app/sitemap.ts` / `app/robots.ts` | 自动生成 sitemap.xml / robots.txt | [app/sitemap.ts](app/sitemap.ts)、[app/robots.ts](app/robots.ts) |
| `app/not-found.tsx` | 自定义 404 | [app/not-found.tsx](app/not-found.tsx) |
| 动态段 `[param]` | 路由参数 | 分页 `[page]`、标签 `[tag]` |
| catch-all `[...slug]` | 匹配多级路径 | 文章详情 [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) |

**两个值得注意的点**：

1. **整个 `app/` 只有一个根布局**——`blog/`、`tags/` 子目录都没有自己的 `layout.tsx`，所有页面共享同一套全局 Shell（Header / Footer / ThemeProvider / ProgressBar）。这意味着导航、主题、进度条是全局一致的。
2. **`pageExtensions` 包含 `mdx`**（[next.config.js](next.config.js) 第 71 行）：`['ts','tsx','js','jsx','md','mdx']`。这让 `.mdx` 文件也能作为路由——不过本项目用 Contentlayer 处理 MDX，并未直接把 `data/blog/*.mdx` 当路由，而是经由 `app/blog/[...slug]` 统一渲染。

### 3.2 Server / Client 组件边界

App Router 的核心心智模型：**默认是 Server Component，只有文件顶部写了 `'use client'` 的才是 Client Component**。

本项目里：

- **Server Component（默认，多数页面）**：首页 [app/page.tsx](app/page.tsx)、文章详情 [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx)、列表、标签、关于、项目页。它们在构建期取数、生成 HTML，**不携带客户端 JS**，是性能与 Lighthouse 满分的基础。
- **Client Component（标了 `'use client'`）**：凡是需要浏览器 API 或交互状态的组件，例如：
  - 主题切换 [components/ThemeSwitch.tsx](components/ThemeSwitch.tsx)（用 `next-themes` 的 `useTheme`）
  - 搜索对话框 [components/search/SearchProvider.tsx](components/search/SearchProvider.tsx)（Context + `useState` + `fetch`）
  - 代码块复制 [components/mdx/Pre.tsx](components/mdx/Pre.tsx)（`navigator.clipboard`）
  - 动画组件（`IntersectionObserver`、`Element.animate`、`window.scrollY`）
  - 评论 [components/comments/GiscusComments.tsx](components/comments/GiscusComments.tsx)（`useEffect` 注入脚本）

**为什么这么分**：把数据获取留在 Server Component（构建期完成、零客户端 JS），只把真正需要交互的部分标为 Client，是 Next.js 性能优化的关键。本项目根布局里**唯一的布局级客户端组件是 `theme-providers`**（[app/theme-providers.tsx](app/theme-providers.tsx)）。

### 3.3 渲染与数据获取策略

| 策略 | 说明 | 本项目用法 |
| --- | --- | --- |
| **SSG（静态生成）** | 构建时预渲染 HTML，默认且主流 | 所有博客页面 |
| **`generateStaticParams`** | 为动态路由**预生成所有路径** | [app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 78 行：遍历 `allBlogs` 把每篇文章的 slug 拆成数组预渲染 |
| **async Server Component 取数** | 在服务端组件里直接 `await` 取数 | 首页 `const sortedPosts = sortPosts(allBlogs)` —— 数据是构建期生成的 `allBlogs`，所以「取数」实际是读静态产物 |
| **`force-static`** | 强制该段静态化 | sitemap / robots / newsletter route 都声明了它 |
| **`output: 'export'`** | 全站纯静态导出 | [next.config.js](next.config.js) 第 57 行，由 `EXPORT=true` 触发 |

> 本项目**没有用 SSR（按请求渲染）和 ISR（增量再生成）**——因为内容全是构建期生成的静态数据，不需要运行时取数。这也是它能静态导出的前提。

### 3.4 Metadata API（SEO 元数据）

App Router 用**导出 `metadata` 对象或 `generateMetadata` 函数**来声明 SEO 元信息，本项目覆盖三层：

1. **根布局静态 `metadata`**（[app/layout.tsx](app/layout.tsx)）：配置 `metadataBase`、`title.template`（`%s | Skyplume Blog`）、`openGraph`、`twitter`、`alternates`（RSS 链接）、`robots`。所有页面继承这些默认值。
2. **页面级辅助函数 `genPageMetadata`**（[app/seo.tsx](app/seo.tsx)）：列表页、关于页等调用它统一生成 OG / Twitter 卡片，缺省值回退到站点默认。
3. **文章级 `generateMetadata`**（[app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 24–76 行）：根据单篇文章动态生成——`openGraph.type` 设为 `'article'`（区别于列表页的 `'website'`），携带 `publishedTime` / `modifiedTime`，`ogImages` 自动检测是否以 `http` 开头来决定拼接 `siteUrl` 前缀。

此外，文章页还在运行时**注入 JSON-LD 结构化数据**（[app/blog/[...slug]/page.tsx](app/blog/%5B...slug%5D/page.tsx) 第 113–116 行），把 `post.structuredData`（构建期算好的 Schema.org `BlogPosting`）用 `<script type="application/ld+json">` 内联，帮助搜索引擎展示文章丰富摘要。

### 3.5 Route Handlers（API 路由）

App Router 里 API 路由写在 `app/.../route.ts`，通过**导出 HTTP 方法函数**（`GET` / `POST` 等）实现。本项目唯一实例 [app/api/newsletter/route.ts](app/api/newsletter/route.ts)：

```ts
export const dynamic = 'force-static'
const handler = NewsletterAPI({ provider: siteMetadata.newsletter.provider })
export { handler as GET, handler as POST }
```

要点：

- 用 `export { handler as GET, handler as POST }` 让同一个函数同时响应 GET 和 POST。
- `export const dynamic = 'force-static'` 是 **Route Segment Config**，强制该路由静态化（详见第 1 章）。
- 想新增真正的运行时接口，就在这里新建 `route.ts`，**不要加 `force-static`**，按需引入数据库即可。

### 3.6 内置优化组件：next/font、next/image、next/link

| 组件 | 本项目用法 | 带来的好处 |
| --- | --- | --- |
| **`next/font`** | [app/layout.tsx](app/layout.tsx) 用 `next/font/google` 加载 Space Grotesk，`display:'swap'`、`preload:true`，通过 CSS 变量 `--font-space-grotesk` 暴露给 Tailwind | 字体自托管、自动子集化、**无 FOUT（字体闪烁）**、不产生第三方请求 |
| **`next/image`** | [components/Image.tsx](components/Image.tsx) 是其薄包装，自动拼 `BASE_PATH` 前缀；MDX 里的本地图片由 remarkImgToJsx 转成 `next/image` 并注入尺寸 | 自动懒加载、按需缩放、避免 CLS |
| **`next/link`** | [components/Link.tsx](components/Link.tsx) 的 CustomLink 对内部链接（`/` 开头）用 `next/link` 实现客户端无刷新路由 | 预取 + SPA 式导航 |

> 静态导出时需 `UNOPTIMIZED=true` 关闭 `next/image` 的内置优化（因为优化服务依赖 Node 运行时），见 [next.config.js](next.config.js) 第 59 行。

### 3.7 next.config.js 关键配置

[next.config.js](next.config.js) 是部署与构建的核心，要点：

| 配置 | 作用 |
| --- | --- |
| `withContentlayer` / `withBundleAnalyzer` 插件组合器 | 用 `plugins.reduce` 把多个 Next 插件叠加（Contentlayer 集成 + `ANALYZE=true` 时启用包分析） |
| `output: process.env.EXPORT ? 'export' : undefined` | 一键切换静态导出 |
| `basePath: process.env.BASE_PATH \|\| undefined` | 子路径部署前缀 |
| `pageExtensions: [..., 'md', 'mdx']` | 让 md/mdx 也可作为页面 |
| `images.remotePatterns`（仅 `picsum.photos`）+ `unoptimized` | 远程图片白名单 + 静态导出时关优化 |
| `headers()` | 注入 7 项**安全响应头**（CSP / HSTS / X-Frame-Options 等），并对 `/static/*` 设一年不可变缓存 |
| `webpack` 钩子 | 加 `@svgr/webpack`，让 SVG 可直接 import 为 React 组件 |
| `reactStrictMode: true` | 开启 React 严格模式 |

### 3.8 静态导出（Static Export）

同一份代码可在「Vercel SSR 部署」与「纯静态导出」间切换，靠三个环境变量：

```bash
EXPORT=true UNOPTIMIZED=true yarn build      # 导出纯静态到 out/
BASE_PATH=/sub yarn build                     # 部署到子路径时加前缀
```

- `EXPORT=true` → `output:'export'`
- `UNOPTIMIZED=true` → 关闭图片优化（静态导出必须）
- `BASE_PATH` → 所有静态资源（社交卡、搜索索引、favicon、RSS）自动加前缀

这是判断「项目无强依赖后端」的有力旁证：能整体静态导出，说明运行期不需要 Node。

### 3.10 与 Contentlayer 的集成

Contentlayer 不是 Next.js 官方组件，但本项目通过 `next-contentlayer2` 的 `withContentlayer` 插件与 Next 深度集成（见 [next.config.js](next.config.js) 第 1、65 行）：

- 在 `next build` / `next dev` 时自动触发 Contentlayer 编译 `data/*.mdx`。
- 生成类型安全的 `allBlogs` / `allAuthors`，通过 [tsconfig.json](tsconfig.json) 的路径别名 `contentlayer/generated` 暴露，在 Server Component 里直接 import 使用。
- `pageExtensions` 含 `mdx` 与 Contentlayer 配合，构成完整的「MDX → 类型安全数据 → 页面」管道。

## 4. 附录

### 4.1 常见疑问 FAQ

**Q1：我部署到 Vercel 之外（比如 GitHub Pages、对象存储）行不行？**
行。用 `EXPORT=true UNOPTIMIZED=true yarn build` 导出纯静态文件即可。子路径部署再加 `BASE_PATH`。

**Q2：评论 / 分析 / 订阅默认能用吗？**
评论默认配置 Giscus 但需填环境变量才能真用（见 [.env.example](.env.example)）；分析组件当前 `return null`（占位）；订阅 API 未接 provider（返回 500 占位）。三者都不影响站点正常启动。

**Q3：我访问草稿文章为什么 404 了？**
`draft: true` 在**生产构建**会被过滤。`yarn dev` 下可见，`yarn build` 后访问会 404。

**Q4：为什么首页只显示 5 篇？**
`POSTS_PER_PAGE = 5`（博客列表），首页 `MAX_DISPLAY = 5`。改这两个常量即可。

**Q5：`og:type` 为什么文章页是 `article`、列表页是 `website`？**
文章页 `generateMetadata` 把 `openGraph.type` 显式设为 `'article'` 并带 `publishedTime`/`modifiedTime`，这是 Google 展示文章丰富摘要的关键信号；列表页用 `genPageMetadata` 默认 `'website'`。

**Q6：MDX 正文里的图片为什么不用手写宽高？**
`remarkImgToJsx` 插件在构建期二进制读取图片文件头拿真实尺寸并注入 `next/image`，既省事又避免布局抖动。
